n-1 & hash的原因

取余运算太慢,位运算更快

n-1 如果=15 高八位全是0，低八位全是1，位运算结果最大值就是 0000 1111 就是15

如果取余，则效率不高

为什么不取余 因为取余本质上也是位运算，但是会多次进行bit位运算，为了避免不必要的开销，所以直接用位运算会更高效

你在使用HashMap的时候，默认赋值初始化的时候，真的是按照你设置的容量进行初始化的吗?

其实不是

```jsx
   @IntrinsicCandidate
    public static int numberOfLeadingZeros(int i) {
        // HD, Count leading 0's
        if (i <= 0)
            return i == 0 ? 32 : 0;
        int n = 31;
        if (i >= 1 << 16) { n -= 16; i >>>= 16; }
        if (i >= 1 <<  8) { n -=  8; i >>>=  8; }
        if (i >= 1 <<  4) { n -=  4; i >>>=  4; }
        if (i >= 1 <<  2) { n -=  2; i >>>=  2; }
        return n - (i >>> 1);
    }
```

为什么红黑树里面需要有双向链表？扩容

binCount= 7 的时候，此时链表中已经有了8个元素，此时执行了赋值.next，所以在判断是否转为红黑树之前，实际链表已经有了9个月元素，也就是实际转的时候是在9个元素的时候转的红黑树

而且转红黑树是只要链表长度大于8就转吗？

根据源码看不是的，是原数组的长度要大于64，并且数组上存在长度大于8的链表，此时链表才会转为红黑树

如果原数组的长度没有达到64，但是存在长度大于8的链表，会进行扩容，扩容会将长链表转为两个短链表，一样会提高效率；因为转红黑树的原因就是链表的查询效率低，越长越低，截断变短效率会变高。树查询的效率是logN

长链表拆成短链表的过程伴随着扩容会进行，根据所有节点二次分布的结果，称之为low and high 高低位置链表，得出最后的结果，可能会变成会产生两个链表，也可能是一个链表一个树等等

所以如果转成红黑树，具体过程做法是：

（1）类型升级：将原有的Node类型转为TreeNode类型；

（2）增加前置指针：变为双向链表；

最后是从双向链表转为红黑树

扩容

（1）分类型

（2）分位置是否有后续next

（3）链表或者树节点迁移需要关注新下标位置

只有两种情况：
